# QuantAgent V7 证据摄取与交易规则 / Evidence Ingestion and Trading Rules

本文档覆盖 V7 中"每日证据摄取层 + 公司暴露映射 + 股票池门槛 + 长期因子自适应权重 + Long-Short 资金分配绑定 + 工具化对冲 + 新闻交叉验证 + 生命周期交易规则"。这些模块共同把 V7 从"研究报告 pipeline"升级到"可执行约束系统"。

## 1. 每日证据摄取层 / Daily Evidence Ingestion Layer

源代码位置：`src/quantagent/data/ingestion/`。

### 统一证据表 / Unified Evidence Frame

所有 ingestor 都必须输出包含 :data:`EVIDENCE_COLUMNS` 的 DataFrame：


### Daily Evidence Job

```python
job = DailyEvidenceJob(
    registry=SourceCredibilityRegistry(),
    ingestors={
        "policy": PolicyIngestor(),
        "disclosure": DisclosureIngestor(),
        "news": NewsIngestor(),
        "financial": FinancialIngestor(),
        "order_contract": OrderContractIngestor(),
        "regulatory_penalty": RegulatoryPenaltyIngestor(),
    },
)
result = job.run(DailyEvidenceJobConfig(as_of_date="2026-05-15"))
```

`result.frame` 是合并后的 PIT-safe 表，结果会被写入 `data/v7/evidence/evidence_<date>.csv`。

## 2. 公司暴露映射 / Company Exposure Mapping

`themes/company_exposure_mapper.py` 已**删除硬编码 alias map**，改为按以下信号判断：

```text
1. 结构化收入字段：{node}_revenue_exposure / revenue_exposure_estimate
2. 订单 / 公告 EvidenceRecord 计数
3. 新闻 EvidenceRecord 计数
4. 来自 IndustryChainReasoner 的动态 node_name / chain_nodes 关键词
5. 公司公告中是否含"占比较小 / 未对公司业绩"等免责声明
```

输出分类：

* `DIRECT_EXPOSURE` — 直接产品收入 ≥ 15% 且有公司证据
* `CRITICAL_BOTTLENECK` — 在 bottleneck 节点上 + 高收入 + 证据
* `UPSTREAM_SUPPLIER` — 有订单证据 + 上下游
* `WEAK_ASSOCIATION` — 仅新闻概念，无收入证明
* `FALSE_ASSOCIATION` — 检测到免责声明 or 仅关键词命中无收入

## 3. 股票池硬门槛 / Stock Pool Hard Gate

`v7/scoring.py::classify_universe_bucket` 增加了硬门槛字段：

```text
CORE:
  exposure_score >= 80
  fundamental_score >= 70
  evidence_count >= 3
  valuation_score >= 35
  fraud_risk_score <= 50
  liquidity_score >= 50
  source_confidence >= 0.65
  revenue_exposure_estimate >= 0.15  (when supplied)
  exposure_type in {direct_exposure, critical_bottleneck, domestic_substitution}

STRONG:
  exposure_score >= 65
  evidence_count >= 2
  fundamental_score >= 50
  fraud_risk_score <= 65

EXCLUSION:
  fraud_risk_score > 80
  liquidity_score < 25
  source_confidence < 0.25
  company_disclaimer_detected and revenue_exposure < 0.05
  exposure_type == false_association
```

新字段是可选的，没有提供时回退到旧版相对宽松的逻辑，避免破坏现有调用方。

## 4. 长周期因子自适应权重 / Adaptive Long-Horizon Factor Weights

`factors/long_horizon_weight_learner.py` 提供 walk-forward IC → 因子权重学习：

```python
learned = learn_long_horizon_weights(
    panel,  # 含 forward_return_120d
    config=WeightLearnerConfig(walk_forward_splits=4, embargo_days=10),
)
weights = select_weights(learned, theme="ai_compute", market_regime="bull")
```

输出 `LearnedWeights`：

```text
global_weights        全局加权
per_theme             {theme: weights}
per_sector            {sector: weights}
per_regime            {market_regime: weights}
per_lifecycle         {lifecycle_stage: weights}
walk_forward_windows  实际窗口数
```

`select_weights` 按 theme → sector → regime → lifecycle → global 优先级返回最具体的权重。窗口或样本不足时回退到均匀先验。

## 5. Long-Short 资金分配绑定 / Bound Sleeve Override

`construct_v7_portfolio(..., sleeve_weights_override=...)` 现在接受一个外部 sleeve 权重 dict。Pipeline 会先跑 `allocate_long_short`，把结果 sleeve_weights 转成 override，传入 portfolio builder：

```python
allocation = allocate_long_short(alphas, members, market, None, cfg)
override = _sleeve_weights_from_long_short(allocation, allocator_cfg)
portfolio = construct_v7_portfolio(..., sleeve_weights_override=override)
```

配置开关：

```yaml
long_short_allocator:
  bind_to_portfolio: true
```

设为 `false` 时退回到 ``_dynamic_sleeves`` 的固定先验。

## 6. 工具化对冲 / Tool-based Hedge

`portfolio/portfolio_beta_model.py` 用 OLS 估算每只票相对沪深 300 的 beta，并加总成 `portfolio_beta`。

`portfolio/hedge_instrument_selector.py` 根据 portfolio_beta、目标 beta、hedge_need_score 选择：

```text
HedgeAction:
  NONE
  REDUCE_GROSS
  CASH_BUFFER
  BROAD_INDEX_HEDGE     CSI300 / CSI500 / CSI1000 ETF
  SECTOR_HEDGE          半导体ETF / AI ETF / 新能源ETF / 军工ETF
  BETA_REDUCTION
  SUSPEND_NEW_OPENS
```

每个 instrument 有 `cost_bps`、`max_notional_weight`、`estimated_beta`。算法是受约束的贪心：先优先减低主敞口的行业 ETF，再用宽基 ETF 补足剩余 beta；当 `hedge_need_score >= 0.45` 强制加 cash buffer；`>= 0.65` 暂停开新仓。

配置：

```yaml
tool_hedge:
  enabled: true
  benchmark_symbol: 000300.SH
  target_beta: 0.30
  max_total_hedge_weight: 0.25
```

## 7. 新闻交叉验证 / News Cross-Validation

`credibility/news_cross_validator.py` 按 `(symbol, theme, event_type)` 汇总：

```text
confirming_sources       不同来源数
primary_confirmations    交易所 / 政府原文确认数
official_confirmations   官方来源数
contradiction_count      正文中检测到澄清 / 否认 / 辟谣
same_source_reposts      同 raw_hash 重复次数
after_close_only         仅盘后发布
rumor_risk               基于关键词："据传 / 据悉 / 未经证实"
most_authoritative_source
```

`attach_cross_validation_fields` 把这些值写回 `NewsCredibilityScore` 的 `cross_validation_count` / `contradiction_flags` / `rumor_risk`，让下游 `news_confidence_score` 重新计算后能反映真实交叉验证情况。

Pipeline 会自动调用：

```python
summaries = cross_validate(bundle.news.frame)
news_scores = attach_cross_validation_fields(news_scores, summaries)
```

## 8. 生命周期交易规则 / Lifecycle Trading Rules

`themes/lifecycle_trading_rules.py` 给每个 `ThemeLifecycleStage` 一条硬规则：

| Stage | 单票上限 | 主题总上限 | 开新仓 | 加仓 | 强制减仓 | 强制清仓 | Sleeve |
| --- | --- | --- | --- | --- | --- | --- | --- |
| POLICY_SEED | 0.00 | 0.00 | ✗ | ✗ | ✗ | ✗ | watchlist |
| NARRATIVE_FORMATION | 0.02 | 0.10 | ✓ | ✗ | ✗ | ✗ | medium_theme |
| CAPITAL_INFLOW | 0.04 | 0.20 | ✓ | ✓ | ✗ | ✗ | medium_theme |
| FUNDAMENTAL_VALIDATION | 0.06 | 0.30 | ✓ | ✓ | ✗ | ✗ | long_fundamental |
| EARNINGS_REALIZATION | 0.08 | 0.35 | ✓ | ✓ | ✗ | ✗ | long_fundamental |
| VALUATION_BUBBLE | 0.03 | 0.15 | ✗ | ✗ | ✓ | ✗ | hedge |
| DIVERGENCE | 0.04 | 0.20 | ✗ | ✗ | ✓ | ✗ | medium_theme |
| DECAY | 0.01 | 0.05 | ✗ | ✗ | ✓ | ✗ | cash_buffer |
| INVALIDATED | 0.00 | 0.00 | ✗ | ✗ | ✗ | ✓ | cash_buffer |

`apply_lifecycle_caps` 在 portfolio 构建之后自动调用，强制每个 target_weight 不超过其主题阶段的上限。Pipeline 输出包含 `lifecycle_caps_applied`，记录每条调整原因。
