## 主题发现算法 / Theme Discovery Algorithm

Theme Discovery 不是新闻热度排序，而是多源证据加权状态机：

```text
theme_strength =
  0.28 * policy_strength
  + 0.20 * market_strength
  + 0.24 * industry_fundamental_strength
  + 0.18 * capital_flow_strength
  + 0.10 * news_sentiment_strength
  - 0.20 * opposing_evidence_penalty
  - 0.10 * bubble_risk
```

生命周期判断：

- `policy_seed_stage`：政策强、市场弱、产业兑现弱。
- `narrative_formation_stage`：政策强，新闻和资金开始形成叙事。
- `capital_inflow_stage`：板块放量、资金流入、龙头走强，但订单和利润尚未验证。
- `fundamental_validation_stage`：订单、营收、产能、价格或供需缺口开始验证。
- `earnings_realization_stage`：利润兑现、估值重定价，基本面证据强。
- `valuation_bubble_stage`：股价和估值涨幅显著领先基本面，crowding 高。
- `divergence_stage`：真龙头和伪概念分化，弱相关股票退潮。
- `decay_stage`：政策、资金、业绩催化衰减。
- `invalidated_stage`：核心假设被证伪或监管风险显著上升。

政策权威分数：

- 中央 / 国务院：0.90-1.00。
- 部委：0.80-0.95。
- 省市：0.60-0.85。
- 行业协会：0.40-0.70。
- 地方园区：0.30-0.60。
- 媒体解读：0.20-0.50。

所有主题都必须保存 `key_evidence`、`opposing_evidence`、`required_follow_up_data` 和 `expiry_date`。超过有效期必须重新验证。

## 产业链图谱算法 / Industry Chain Graph Algorithm

Industry Chain Graph Agent 使用三步构建：

1. Seed expansion：从政策关键词、行业报告、公告业务、产品名和供应链关系扩展 chain node。
2. Relation classification：把节点关系标为 `direct_exposure`、`critical_bottleneck`、`upstream_supplier`、`downstream_application`、`infrastructure_dependency`、`cost_beneficiary`、`domestic_substitution`、`capacity_expansion`、`customer_supplier_link`、`technology_enabler`、`policy_beneficiary`、`weak_association`、`false_association`。
3. Benefit validation：判断受益来源是 revenue growth、margin expansion、valuation repair、cost reduction、policy subsidy、domestic substitution，还是纯短期 sentiment。

每个 chain node 至少计算：

- `dependency_strength`：对主题兑现的依赖强度。
- `bottleneck_score`：是否是瓶颈环节。
- `domestic_substitution_score`：国产替代弹性。
- `supply_shortage_score`：供需错配。
- `price_elasticity` 和 `profit_elasticity`。
- `demand_visibility`：订单、客户、政策采购、下游需求证据。
- `policy_support_score`。
- `technology_barrier` 和 `competition_intensity`。

伪相关识别规则：

- 公司公告没有主题相关收入或订单，且只有新闻标题关联。
- 主题相关业务收入占比低于阈值，且无法验证利润弹性。
- 股价涨幅大，但公告反复风险提示“业务占比较小”。
- 供应链关系只有二手传闻，无客户或订单交叉验证。

## 动态股票池算法 / Dynamic Thematic Universe

股票池构建顺序：

```text
Base A-share Universe
-> tradability filter
-> ST / suspension / delisting risk filter
-> liquidity filter
-> sector and industry mapping
-> theme mapping
-> chain node mapping
-> true / strong / weak / false association
-> fundamental investability filter
-> fraud, valuation, crowding and liquidity risk filter
-> dynamic thematic universe
```

分层规则：

- Core Beneficiary Pool：`exposure_score >= 80`、`fundamental_score >= 70`、`evidence_count >= 3`、`fraud_risk_score <= 60`、流动性合格。
- Strong Correlation Pool：强相关但仍需验证收入或利润兑现。
- Optional Satellite Pool：间接受益、弹性大、风险也大，只允许小仓位。
- Watchlist Pool：逻辑可能成立，但证据不足。
- Exclusion Pool：伪概念、弱相关、财务风险高、流动性差、估值透支。

短线主题股票池有效期 1D-20D，波段 20D-60D，中线 60D-120D，最长观察期 126D。过期后必须重新跑 Theme Discovery 和 Fundamental Validation。

## 基本面评分 / Fundamental Scoring

`fundamental_score` 建议采用 0-100：

```text
fundamental_score =
  0.18 * business_model_fit
  + 0.18 * revenue_exposure
  + 0.14 * profit_elasticity
  + 0.12 * order_capacity_visibility
  + 0.14 * financial_quality
  + 0.10 * governance_quality
  + 0.10 * valuation_margin_of_safety
  + 0.04 * disclosure_consistency
```

中线或长期股票池必须额外满足：

- 主题相关收入、订单、产能或客户有可验证证据。
- 经营现金流和利润没有长期严重背离。
- 估值没有明显透支 120D 兑现路径。
- 没有重大 fraud risk、监管处罚或非标审计。

## 财务造假风险评分 / Financial Fraud Risk

`Financial Fraud Risk Agent` 使用强制折减规则：

- `fraud_risk_score > 80`：禁止进入长期股票池，正向基本面信号 confidence 乘以 0.2，新闻利好 confidence 乘以 0.3，标记 `high_fraud_risk`。
- `60 <= fraud_risk_score <= 80`：降低长期持仓上限，正向信号 confidence 乘以 0.5，需要更多证据验证。
- 曾经重大造假：默认不进入 Core Beneficiary Pool，除非长期修复证据明确。

指标集合：

- Beneish M-score。
- Piotroski F-score。
- Altman Z-score。
- accruals quality。
- cashflow quality。
- receivables risk。
- inventory risk。
- related-party risk。
- regulatory penalty score。
- audit opinion score。

## 新闻可信度评分 / News Credibility

News Credibility 不把标题直接变成交易信号。建议公式：

```text
news_confidence =
  0.55 * source_reliability
  + 0.15 * is_primary_source
  + 0.15 * is_official
  + min(0.15, 0.04 * cross_validation_count)
  - min(0.25, 0.08 * contradiction_count)
  - 0.25 * rumor_risk
```

周期分类：

- 交易性新闻：1D-5D。
- 事件催化新闻：5D-20D。
- 行业趋势新闻：20D-60D。
- 政策兑现新闻：60D-120D。
- 无法验证新闻：只进观察，不进交易 signal。

新闻必须输出 `contradiction_flags` 和 `rumor_risk`。公司公告、交易所披露、政策原文优先；自媒体、论坛、二手转载必须降权。

## 多周期 Alpha / Multi-Horizon Alpha

V7 输出 `alpha_1d`、`alpha_5d`、`alpha_20d`、`alpha_60d`、`alpha_120d`、`alpha_126d`：

- 1D：技术指标、资金流、隔夜新闻、涨跌停风险、波动率。
- 5D：短线事件、新闻情绪、主力资金、趋势延续。
- 20D：行业轮动、板块强弱、财报预期、主题扩散。
- 60D：订单、产能、行业景气、政策执行、估值修复。
- 120D / 126D：政策周期、产业趋势、国产替代、盈利兑现、行业供需和估值成长匹配。

输出必须包括：

- expected return 和 expected excess return。
- volatility forecast 和 downside risk。
- conformal confidence。
- prediction interval。
- factor contribution 和 evidence contribution。
- risk penalty。
- final alpha score。

仲裁规则：

- 短周期强、长周期弱：只能 short_event sleeve，小仓位。
- 长周期强、短周期过热：进入 watchlist 或等待回调。
- 长周期强、短周期低位启动：可进入 medium_theme 或 long_fundamental。
- 预测区间过宽：降低 target weight。
- 模型漂移高：降低所有风险资产权重。

## 因子适用域 / Factor Applicability

每个 factor 必须绑定：

- applicable universe。
- applicable sector。
- applicable theme。
- applicable market regime。
- horizon days。
- decay half life。
- RankIC、ICIR、hit rate、turnover、capacity、crowding。
- lifecycle stage：discovery、validation、production、crowded、decaying、invalidated。

使用规则：

- 短线技术因子不能用于 60D / 120D 基本面持仓。
- 行业专用 factor 不能泛化到全市场。
- crowding 高或 RankICIR 失效时，factor 降权或改作 filter。
- 因子过期后必须重新 walk-forward 验证。

实现细节：

- `factor_applicability_agent` 现在用 `member.sector`（真实行业分类）做 sector slice，不再把 `chain_node` 错当 sector，sector applicability 与 chain_node applicability 分别评估。
- 默认 alpha 模型由 `v7_classical_alpha.py` 提供 Ridge / ElasticNet 多周期回归，按 `forward_return_{horizon}d` 列做 walk-forward 训练。需要时可切回 `v7_deep_alpha.py`（多塔 deep alpha）。

## 组合仓位分配 / Portfolio Construction

V7 portfolio 使用 sleeve allocation：

```text
long_fundamental
medium_theme
short_event
sector_rotation
hedge
cash_buffer
```

默认逻辑：

- 市场强、主题强、基本面强：提高 `medium_theme` 和 `long_fundamental`。
- 市场弱、主题强：降低 gross exposure，保留核心，增加 cash buffer。
- 市场弱、主题弱：高现金或对冲。
- 主题强但估值泡沫：降低仓位，只做择时或等待。
- 短线情绪强但基本面弱：只允许 `short_event` 小仓位。
- 财务风险高：禁止 `long_fundamental`。
- 流动性差：降低 target weight。
- prediction interval 过宽：降低 target weight。

Portfolio Construction Agent 的输出只能是 `PortfolioPlan` 和 `target_weights`，不能输出 broker order。

## 对冲决策 / Hedge Decision

A 股散户的 hedge 不等于必须使用复杂衍生品。可用动作：

- 降低总仓位。
- 增加现金。
- 降低单主题集中度。
- 降低高 beta 股票。
- 增加低波动或防御资产。
- 合法可行时使用宽基 / 行业反向工具。
- 行业分散。
- 暂停短线交易。

触发条件：

- market regime 进入 `risk_off`。
- 市场宽度恶化。
- 板块退潮。
- 组合回撤超过阈值。
- 波动率快速上升。
- 主题 crowding 过高。
- 龙头跌破关键位置。
- 资金明显流出。
- fraud / news / model uncertainty 上升。
- 单一主题暴露过大。

**散户对冲可执行性 / Retail Hedge Feasibility**

`portfolio/retail_hedge_feasibility.py` 在工具型 hedge recommendation 后再过一道：

```python
result = filter_recommendation_for_retail(
    recommendation,
    RetailAccountCapabilities(
        can_short_individual_stock=False,
        can_short_index_futures=False,
        can_trade_options=False,
        can_buy_etf_inverse=True,
        can_hold_cash=True,
        can_reduce_gross_exposure=True,
    ),
    instrument_universe=instruments,
)
```

行为：

- 不允许做空个股 / 期货 / 期权时，对应 action 被剥除；
- 反向 ETF 不可买入时，对应 instrument weight 被剥除并重新计入 cash buffer；
- 当所有可执行动作都被屏蔽时，最终回退到 `CASH_BUFFER` 单一动作；
- 输出 `feasibility_score`（被允许的 action 占总 action 比例）和 `audit_notes`，写入 `retail_hedge_feasibility` 字段供审计。

策略报告中“理论可行但实操不可执行”的 hedge 必须由此组件过滤后再进入执行模拟。

## 风控规则 / Risk Gate Rules

Risk Gate 先于任何 execution preparation：

- 单票最大权重、最大亏损、流动性、财务风险、fraud risk。
- 行业最大权重、相关性、crowding、退潮风险。
- 单主题最大权重、生命周期、bubble risk、invalidation risk。
- 总仓位、净敞口、毛敞口、波动率、VaR、CVaR、max drawdown、turnover、model uncertainty。
- 数据缺失、延迟、质量差、future leakage、未确认新闻、未经验证财报。
- 模型漂移、factor invalidation、prediction interval 过宽、样本外失效、backtest overfitting。

输出：

- `risk_passed`。
- `rejected_symbols`。
- `reduced_symbols`。
- `blocked_symbols`。
- `risk_warnings`。
- `max_allowed_position`。
- `required_cash_buffer`。
- `kill_switch_triggered`。
- `rationale`。

Backtest & Attribution Agent 必须支持：

- walk-forward backtest。
- purged cross validation。
- embargo。
- 每个 factor applicability 输出 `validation_sample_count`、`validation_symbol_count`、`validation_date_count` 和 `walk_forward_pass_rate`；样本不足或 walk-forward 不通过时只能停留在 validation，不得进入 production。
- Point-in-Time data。
- transaction cost、slippage、impact。
- T+1、涨跌停、停牌、流动性约束。
- portfolio rebalance。
- horizon attribution。
- theme attribution。
- sector attribution。
- factor attribution。
- agent attribution。
- risk and hedge attribution。

评价指标：

- annual return、cumulative return、Sharpe、Sortino、max drawdown、Calmar。
- volatility、hit rate、win/loss ratio、turnover、transaction cost。
- alpha、beta、information ratio。
- RankIC、ICIR、factor decay、capacity、tail risk、drawdown recovery days。

必须回答：

- 收益来自哪个 theme、sector、factor、agent。

- 哪些 Agent 有贡献，哪些 Agent 误导。
- 哪些股票是 false association。

**Full-pipeline PIT 回测 / Full-pipeline PIT Backtester**

`backtest/full_pipeline_backtester.py` 是单日级 wrapper 之外的全链路重放：

```python
result = run_full_pipeline_backtest(
    dates,
    price_panel,
    daily_step=lambda as_of: run_daily_v7_research(cfg, as_of_date=as_of),
    config=FullPipelineBacktestConfig(
        initial_capital=1_000_000.0,
        execution_lag_days=1,
        cost_bps=8.0,
        max_single_name_weight=0.06,
    ),
)
```

- `daily_step` 返回 `target_weights` 与 PIT 友好的 audit dict，按日期滚动调用；
- `execution_lag_days` 默认 1 = T+1 fill；当日 target 在下一交易日才被认为是 realised；
- `cost_bps` 按 turnover 收取交易成本；
- `max_single_name_weight` 在 fill 之前再做一次硬截断；
- 输出 `nav` / `daily_returns` / `target_weight_history` / `realized_weight_history` / `universe_size_history` / `pit_audit`。

`build_pit_evidence_slice(frame, as_of_date)` 是 backtester 内部 / 外部都可用的 PIT 切片函数，按 `available_at <= as_of_date` 丢弃未来证据；任何向 `daily_step` 注入数据的层都必须先经过它。

## 因子硬门槛 / Factor Applicability Hard Gate

`factor_applicability.hard_gate=true` 时，未达到 `production` 或 `validation` 阶段的因子 **不会** 进入 V7 Deep Alpha 模型，也不会进入 stock pool selector 的 `applicable_factor_names`。这把 `validate_factor_applicability` 从"诊断报告"升级成"准入门槛"：

```yaml
factor_applicability:
  hard_gate: true
  production_stages: [production, validation]
```

`production_stages` 可以放宽到包含 `insufficient_sample`（用于早期主题或新主题，例如 mock 配置）。`invalidated / decaying / crowded` 的因子永远不会被采纳。

## 学习型 Sleeve 分配 / Walk-Forward Sleeve Allocator

`portfolio/walk_forward_sleeve_allocator.py` 在过去的 sleeve 日收益面板上做 walk-forward 网格搜索：

```yaml
walk_forward_sleeve_allocator:
  enabled: true
  walk_forward_splits: 4
  embargo_days: 5
  min_window_days: 40
  grid_step: 0.05
  drawdown_penalty: 0.50
  cash_floor: 0.10
```

- 每个 walk-forward 窗口内对所有合法 sleeve 权重组合（按 `grid_step` 离散）评分：`sharpe - drawdown_penalty * max_drawdown`。
- 窗口间用 `embargo_days` 隔离防止泄漏。
- 最终输出对各窗口最优解做 score-weighted average，得到 long / medium / short / hedge / cash 的推荐权重。
- 没有历史时退化到 `SleeveConfig.default_weight`，并把 `reason` 标记为 `no_sleeve_return_history` 之类的诊断字符串。

CLI 入口：

```powershell
quantagent walk-forward-v7 --sleeve-returns reports/v7/sleeve_returns.csv --splits 4 --embargo-days 5
```

## 验收标准 / Acceptance Criteria

合格的 V7 系统必须满足：

1. 自动识别政策与产业风口。
2. 判断 theme lifecycle。
3. 构建 industry chain graph。
4. 判断真相关、强相关、弱相关、false association。
5. 构建 dynamic thematic universe。
6. 进行 fundamental due diligence。
7. 识别 fraud risk 和 disclosure credibility risk。
8. 区分 news sentiment 和 policy medium-term signal。
9. 输出 1D、5D、20D、60D、120D、126D alpha。
10. 为 factor 绑定 applicability domain 和 expiry。
11. 结合 market regime 和 sector regime 决定 exposure。
12. 动态分配 sleeves。
13. 判断 hedge need。
14. 模拟 A 股散户执行约束。
15. 进行 Point-in-Time backtest。
16. 输出 theme、sector、factor、agent、risk 和 hedge attribution。
17. 输出 structured evidence 和 audit log。
18. 信息不足时拒绝过度判断。
19. 风险过高时自动降权、减仓、对冲或禁止交易。
20. 每日重新验证 theme、universe、fundamental、risk 和 model validity。
